# 个人工作记忆与周报自动化系统 交接文档

## 一、项目核心背景与定位
### 1.1 根痛点
高强度碎片化工作场景下，人工记忆回溯成本极高，无法完整复盘周期内的工作内容；周报、汇报、复盘等产出均为该核心需求的衍生场景，而非项目本质目标。

### 1.2 产品定位
以「低摩擦工作记忆沉淀」为核心底座，基于统一素材池自动衍生周报、汇报、复盘等多形态输出的个人效率工具。
- **底层核心**：工作记忆采集与沉淀（自动采集 + 手动补录），解决「记不住做了什么」的本质问题
- **上层衍生**：模板化自动输出 + AI 增强，解决「组织表达、多版本重复产出」的效率问题

### 1.3 贯穿全项目的设计原则
1.  **最小化落地**：分阶段迭代，每阶段独立可验收，拒绝过度设计与过早优化
2.  **预留不实现**：关键扩展位提前预埋，当前阶段不做冗余开发，无技术债
3.  **单一数据源**：事件素材为唯一真相源，多视图、多模板均基于同一份数据渲染，避免数据冗余与一致性问题
4.  **体验极简**：用户侧仅保留核心操作入口，底层配置对用户透明，无额外学习成本

## 三、各阶段详细决策与落地方案
### 3.1 P0：模板标准化底座
- **数据层**：复用模板表核心字段，新增 `config` JSON 字段作为通用扩展位，无额外表结构改动
- **官方模板**：内置 4 套标准模板，覆盖主流场景：通用正式、技术研发、极简快速、个人复盘
- **核心能力**：新建周报可选择模板带入内容骨架，完全兼容现有 Milkdown 编辑器

### 3.2 P1：模板动态变量系统
- **变量语法**：采用 `{{变量名}}` 格式，基于正则替换实现，轻量无依赖
- **实现架构**：单文件最小化实现，封装单一 `renderTemplate` 函数；内部通过变量配置表管理，不做多文件模块化拆分，避免过度设计
- **变量范围**：
  - 系统变量：本周日期范围、当前周数、年份、月份
  - 区块变量：核心成果、问题与风险、日常事务、下周计划
- **渲染时机**：服务端统一渲染，两个核心入口
  1.  新建周报：`POST /api/reports` 渲染后将完整内容入库
  2.  模板预览：`GET /api/templates/[id]/render` 实时返回渲染结果，不创建数据库记录

### 3.3 P2：事件素材自动填充模板
- **核心决策**：采用固定 4 种标准 `sectionType`（achievement/risk/routine/plan），所有模板通用
  - 决策依据：将「事件分类属性」与「模板展示名称」解耦，保障一套素材可跨模板复用，符合单一数据源原则
- **渲染规则**：事件按 `sectionType` 自动填充到对应模板变量位置，未标记分类的事件默认归入「日常事务」
- **配置预留**：模板 `config` 字段提前预留区块渲染规则（`maxItems` / `autoSort` / `filterTrivial`），为后续多视图能力铺路

### 3.4 P3：AI智能增强与句式片段库
#### AI 风格体系
- **绑定方式**：模板携带默认 AI 风格，编辑器内支持手动覆盖
  - 优先级规则：前端传入参数 > 模板默认风格 > 全局默认
- **风格定义**：4 种预设风格标签，底层映射独立 Prompt 与温度参数
  - `formal`（正式汇报）、`technical`（技术研发）、`concise`（极简干练）、`detailed`（深度复盘）
- **评分适配**：保留原有三维度评分逻辑不变，根据风格动态调整各维度权重；零重构复用现有评分体系，感知差异明显

#### 句式片段库
- **存储设计**：独立 `sentence_snippets` 表，内置高频预设句式，支持用户自定义新增
- **交互设计**：所有操作收敛于编辑器侧边栏，不做独立管理页面；支持一键插入、选中保存、快捷删除

### 3.5 P4：双视图多渲染引擎
- **核心决策**：采用全维度组合差异（内容筛选 + 表达风格 + 模板结构），而非单一维度差异化
- **交互方式**：编辑器内实时切换预览，数据库仅存一份完整周报数据，视图切换为纯渲染态，不生成冗余记录
- **区块组成**：
  - **领导版**：核心成果、问题与风险、下周计划（3 个区块，剔除日常事务）
  - **个人版**：核心成果、问题与风险、日常事务、下周计划、踩坑记录、问题沉淀（6 个区块）
- **渲染配置**：
  - 领导版：`maxItems=3`、`filterTrivial=true`、`autoSort=true`，高信息密度
  - 个人版：无条数限制、不过滤、不强制排序，完整保留原始内容
- **风格映射**：领导版默认 `formal`（正式汇报），个人版默认 `detailed`（深度复盘）
- **落地方案**：在模板 `config.viewConfigs` 中存储视图配置，官方模板预设默认值；P4 阶段暂不开放用户自定义配置界面，兼顾架构扩展性与落地效率

### 3.6 待落地：Memo 工作记忆模块
- **核心定位**：类浮墨 / Blinko 的低摩擦随手记，作为 Git 自动采集的补充，完善记忆沉淀层
- **数据层设计**：完全复用 `raw_events` 表，新增 `source=manual` 标记来源，不新建独立表，全链路能力天然复用
- **核心交互设计**：
  1.  页面顶部常驻单行输入框，回车即保存，零额外操作
  2.  时间线视图：支持按日 / 周 / 月聚合展示
  3.  快捷分类：支持 `#标签` 语法快速标记分类，默认归入日常事务
  4.  全局快捷键唤起悬浮输入框，不打断工作流
- **能力打通**：所有手动记录自动进入素材池，生成周报时与 Git 采集内容合并渲染

## 四、核心数据模型
基于 SQLite + Drizzle ORM，核心表结构如下：

### 4.1 templates（模板表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | integer | 主键，自增 |
| name | text | 模板名称 |
| content | text | 模板 Markdown 内容 |
| description | text | 模板描述 |
| tags | text | 模板标签 |
| sourceTemplateId | text | 源模板 ID（用于克隆模板） |
| aiStyle | text | 默认 AI 风格：`formal` / `technical` / `concise` / `detailed` |
| config | text(JSON) | 扩展配置，含 `sectionTypeMap`、`viewConfigs` 等字段 |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |

### 4.2 reports（周报表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | integer | 主键，自增 |
| title | text | 周报标题 |
| content | text | 周报 Markdown 内容 |
| weekStart | text | 周起始日期（格式：yyyy-MM-dd） |
| weekEnd | text | 周结束日期（格式：yyyy-MM-dd） |
| aiStyleOverride | text | AI 风格覆盖（可选） |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |

### 4.3 raw_events（事件素材表）
承载 Git 自动采集与手动 Memo 两类素材，是全系统的唯一真相源
| 字段 | 类型 | 说明 |
|------|------|------|
| id | integer | 主键，自增 |
| eventTime | timestamp | 事件发生时间 |
| source | text | 来源标识（采集源名称） |
| content | text | 事件内容描述 |
| metadata | text(JSON) | 元数据（含 `sha`、`url`、`repo` 等 Git 信息） |
| category | text | 用户自定义分类 |
| sectionType | text | 系统分类枚举：`achievement` / `risk` / `routine` / `plan`，默认 `routine` |
| status | text | 状态：`pending` / `processed`，默认 `pending` |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |

### 4.4 sentence_snippets（句式片段表）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | integer | 主键，自增 |
| content | text | 句式内容 |
| category | text | 分类，默认 `通用` |
| isBuiltIn | boolean | 是否为官方内置，默认 `false` |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |

### 4.5 collect_sources（采集源表）
管理 Git 数据采集配置
| 字段 | 类型 | 说明 |
|------|------|------|
| id | integer | 主键，自增 |
| type | text | 采集源类型：`github` / `gitlab` / `local-git` |
| name | text | 采集源名称 |
| config | text(JSON) | 采集配置（含 `baseUrl`、`owner`、`repo`、`token`、`authorEmails` 等） |
| enabled | boolean | 是否启用，默认 `true` |
| lastSyncAt | timestamp | 最后同步时间 |
| lastSyncStatus | text | 最后同步状态 |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |

## 五、技术栈说明
- 前后端框架：Next.js 16（App Router 模式）
- 开发语言：TypeScript
- 数据库：SQLite
- ORM：Drizzle ORM
- 富文本编辑器：Milkdown
- AI 能力：OpenAI 兼容 API（可配置 API URL 和模型）
- 样式方案：Tailwind CSS
